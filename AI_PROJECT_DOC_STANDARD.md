# AI Project Documentation Standard (External + Developer)

> 適用範圍：所有對外發布的 AI 專案 README / 官網技術文件。

## A. 文件整體原則
1. **前商業、後技術**：前段給採購與決策者，後段給開發者。
2. **可驗證**：所有「可運作」描述須對應可重現的驗證步驟。
3. **可擴充**：硬體驗證區塊必須可持續新增平台，不可寫死單一型號。
4. **圖像化成果**：至少一張執行結果圖（或 GIF），避免只有文字敘述。

---

## B. 固定章節順序（必須）

### 1) 專案標題與一句話定位
- 專案名稱
- 一句話價值主張（例如：即時、可部署、可維運）

### 2) 專案資訊
- 專案定位與可解決問題
- 導入價值（導入成本、風險、時程、維運）
- 若有平台策略，需清楚說明

### 3) Edge_AI_SDK / InferenceKit 前置價值（若適用）
- 放置連結：<https://ess-wiki.advantech.com.tw/view/Edge_AI_SDK/InferenceKit>
- 說明「先安裝 InferenceKit，再安裝專案」的好處
- 強調開發者不必自行處理底層環境

### 4) 已驗證硬體平台（可擴充表格）
- 欄位至少包含：平台、驗證狀態、連結、備註
- 第一列填當前已驗證硬體
- 必須保留「待新增平台」列

### 5) 執行結果示意（圖片/GIF）
- 至少 1 張結果圖
- 建議 2 張（即時畫面 + 推論輸出）

### 6) 開發者資訊（安裝與執行）
- 前置條件
- 安裝步驟
- 第一次啟動
- 第二次啟動（快取重用）
- 離線啟動（若支援）
- 驗證方式（health check / output 檔案）
- 常見問題

### 7) 規範引用
- 文件結尾需指向本規範，宣告後續專案遵循

---

## C. 撰寫格式規範
1. 指令區塊必須可直接複製執行。
2. 版本與平台資訊需明確（如 JetPack / CUDA / TensorRT）。
3. 避免模糊字詞（例如「應該可以」），改成可驗證描述。
4. 連結使用完整 URL。
5. 圖片路徑需為 repo 內有效相對路徑。
